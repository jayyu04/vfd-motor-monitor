02 系統架構
十二個模組，每個只做一件事

系統採用嚴格分層設計，每個模組職責單一，不依賴其他模組的內部狀態。未來換真實硬體只需修改 comms.py 一個模組，換不同規格的馬達只需修改 config.py，其他模組完全不動。

config.py馬達銘牌規格、系統參數、Rules 閾值的集中管理，所有模組都來這裡讀取設定
records.py定義 FullRecord 資料結構，存放合併後的完整紀錄，避免 main.py 和 database.py 循環引用
VFD_simulator.py模擬真實變頻器，只輸出三個原始訊號：Hz / A / V，同時提供 get_current_fault() 讓 dashboard 即時讀取目前模擬的工況
comms.py通訊層，VFD 與電腦的橋樑。目前從 simulator 讀取，未來換 Modbus 只改這裡
data_collector.py接收原始訊號，加上 timestamp 和 motor_id，完成資料擷取
physics.py用物理公式從三個原始值推算：同步轉速、轉差率、轉矩
rules.py規則式分析層，計算 Anomaly Score（0~100），第一層判斷，結果回傳給 main.py
ml_model.py隨機森林機器學習，100棵決策樹投票，第二層判斷，結果回傳給 main.py
main.py流水線串接，判斷啟動遮蔽，合併 Rules + ML 結果，寫入資料庫
control.py系統狀態唯一管理者，接收 dashboard 指令，通知 main 開始/停止
database.pySQLite 資料庫，儲存完整紀錄，提供查詢介面給 dashboard
dashboard.py展示層，只跟 control.py 說話傳送指令，從 database.py 讀取資料顯示，從 VFD_simulator 讀取實際工況

04 VFD_simulator — 模擬資料產生
為什麼需要模擬器

真實工業監控需要接實體感測器，但開發和展示階段沒有真實硬體。VFD_simulator.py 的任務是產生「看起來像真實變頻器輸出」的資料，只輸出三個原始值：Hz / A / V，模擬真實硬體透過 Modbus 能直接讀到的最小資料集。

連續性設計：資料不會一下高一下低

真實感測器的數值不會每秒大幅跳動，而是在某個中心值附近小幅浮動。simulator 用「滑動狀態」實現這個特性：

第一筆：從工況的中心值範圍隨機取初始值
之後每筆：上一筆數值 + 小幅隨機漂移

# 範例：NORMAL 模式，頻率中心值 47.3Hz
47.3 → 47.5 → 47.2 → 47.4 Hz（平滑連續，不跳動）
三種運作狀態
OFF 狀態：
Hz=0 / A=0 / V=0，重置所有內部狀態

STARTUP（開機後 0~5 秒）：
頻率從 5Hz 線性爬升到目標頻率
電流是額定的 1.5~2 倍（22~30A）← 變頻器限流啟動，正常現象
隨啟動完成線性降回正常範圍

RUNNING（開機 5 秒後）：
依選擇的工況產生對應的穩態資料
六種工況設計

五種手動模式 + 一種自動模式，各模式的頻率和電流範圍刻意設計了重疊區間，讓 ML 能學到模糊邊界。

NORMAL　正常運轉
頻率：30~60 Hz（操作員設定範圍）
電流：10.5~14 A
頻率由操作員決定，電流反映負載大小，兩者獨立，符合 V/f 控制
操作範圍 30~60Hz 由從業人員確認，電流 10.5~14A 對應 70%~93% 額定負載
OVERLOAD　過電流
頻率：30~60 Hz（任何操作頻率）
電流：18.5~22 A（110%+ FLA）
任何操作頻率下電流超過額定值，模擬機械過載。頻率不侷限於特定範圍，因為過載可以在任何轉速下發生
參考依據：IEC 60947 過電流保護，觸發門檻 110% FLA（16.5A）
BEARING_WEAR　軸承磨損
頻率：30~60 Hz（任何操作頻率）
電流：10~13 A（正常，但波動 ±2.5A）
任何操作頻率下電流中心值正常但波動幅度大，模擬軸承磨損產生的電流漣波
參考依據：ISO 10816 振動監測標準（真實需振動感測器）。本系統以電流標準差作為簡化替代指標
LOAD_LOSS　負載流失
頻率：30~60 Hz（任何操作頻率）
電流：2~4 A（空載以下）
任何操作頻率下電流極低，模擬皮帶斷裂或幫浦空轉。負載流失可以在任何轉速下發生
參考依據：幫浦空轉保護（Dry Run Protection）業界慣例，電流低於正常負載電流判定為負載流失
STALL　機械卡死
頻率：20~35 Hz（拉不上去）
電流：24~30 A（160%+ FLA）
電流飆高且頻率拉不上去，模擬機械端堵轉
參考依據：IEC 60947 堵轉保護，觸發門檻 133% FLA
AUTO　自動隨機切換
每 10~20 秒自動切換工況（不重複上一個）
系統自動切換工況，Rules 和 ML 在不知道工況的情況下自己判斷。VFD Header 即時顯示目前實際工況，趨勢圖下方出現對比表格，可以即時驗證 ML 準不準確。這是展示 ML 真實能力最有說服力的方式。
05 物理計算層 — 從三個原始值推算物理量
為什麼要推算

真實變頻器透過 Modbus 只給出頻率、電流、電壓三個值，但監控和診斷需要更多有意義的物理量。physics.py 是系統裡唯一包含物理公式的地方，所有推算都在這裡完成。

三個核心公式
# 1. 同步轉速（理論最高轉速）
sync_rpm = 120 × frequency_hz / 極數
　　　　　= 120 × 50 / 4 = 1500 RPM（額定頻率下）
　→ 120 是電氣常數，極數決定轉速，極數越多轉速越慢

# 2. 轉差率（從電流負載比估算）
load_ratio = current_a / RATED_CURRENT_A
　→ 實際電流佔額定電流的比例，代表目前負載程度
slip_ratio = 0.02 + load_ratio × 0.05
　→ 負載越重，轉差率越大，轉速越慢
　→ 0.02 是空載時的基本轉差，× 0.05 是負載影響係數
　※ 沒有轉速感測器，用電流近似估算，是估計值

# 3. 轉矩（理論值）
P_in = √3 × V × I × cosφ
　→ 三相電功率公式，√3 是三相電的幾何係數，cosφ 是功率因數
P_out = P_in × η
　→ 輸入功率乘以效率，得到實際輸出的機械功率
ω = 2π × sync_rpm / 60
　→ 將每分鐘轉速（RPM）換算成每秒弧度（角速度）
T = P_out / ω
　→ 功率 ÷ 角速度 = 轉矩，單位 N·m（牛頓米）
直接量測值（可信）

frequency_hz
current_a
voltage_v

這三個直接來自感測器，最可靠。

估算值（理論值）

sync_rpm
slip_ratio
torque_nm

用公式推算，沒有轉速感測器，屬於理論估計值。

06 Rules 判斷 — 規則式異常評分
核心概念：用分數取代是非題

傳統做法是直接判斷「正常」或「異常」，沒有灰色地帶。我們改用 Anomaly Score（0~100分），分數越高代表越嚴重，讓判斷有層次感。操作員不只知道「有沒有問題」，還能看到「分數在慢慢爬升」，提早做出應對。

白話比喻
就像體溫計，不只告訴你「有沒有發燒」，而是告訴你「現在 38.5°C，比昨天的 37.8°C 高了」。分數的變化趨勢比單純的是非判斷更有價值。

四個條件如何計分

四個條件各自獨立計算分數，取分數最高的那個作為最終結果，不累加。這樣設計確保各模式的嚴重程度不會互相干擾。

STALL（機械卡死） → 最高 100 分 → 漸進式 DANGER / CRITICAL

觸發條件：電流 > 20A（133% FLA，IEC 60947）
計分公式：score = (電流 - 20) × 10，上限 100

電流 22A → 20分 → NORMAL　電流 25A → 50分 → DANGER
電流 28A → 80分 → CRITICAL　電流 30A → 100分 → CRITICAL

為什麼這樣設計：STALL 是漸進式的，電流剛超過 133% FLA 可能只是暫時性過載，電流越高才越確定是卡死。漸進分數讓操作員能觀察到惡化趨勢，不是一超過門檻就直接 CRITICAL。模糊地帶（20~22A）由 ML 負責判斷是 OVERLOAD 還是 STALL。

LOAD_LOSS（負載流失） → 最高 74 分 → 永遠 DANGER

觸發條件：電流 < 5A
計分公式：score = (5 - 電流) / 5 × 24 + 50，上限 74

電流 4.9A → 50分 → DANGER 下緣　電流 3A → 59分 → DANGER
電流 1A → 69分 → DANGER　電流 0A → 74分 → DANGER 上緣

為什麼這樣設計：負載流失代表皮帶斷裂或幫浦空轉，馬達在空轉毫無產出，屬於結構性異常，一旦發生就是危險，所以分數從 DANGER 下緣（50分）開始，不給 WARNING 緩衝。電流越低代表越確定是空轉，分數越高。

OVERLOAD（過電流） → 最高 49 分 → 永遠 WARNING

觸發條件：電流 > 16.5A（110% FLA，IEC 60947）
計分公式：score = (電流 - 16.5) × 9，上限 49

電流 17A → 4分 → NORMAL　電流 20A → 31分 → WARNING
電流 21A → 40分 → WARNING　電流 22A → 49分 → WARNING 上緣

為什麼這樣設計：過電流是馬達可以短暫承受的狀態，不是立即危險。設計上限 49 分確保 OVERLOAD 永遠在 WARNING，提醒操作員注意但不造成恐慌。輕度過電流（17~19A）分數落在 NORMAL，由 ML 補位，反映了「剛超過門檻不一定是異常」的現實。

BEARING_WEAR（軸承磨損） → 最高 49 分 → 永遠 WARNING

觸發條件：電流標準差 > 0.8A 且 頻率變化 < 0.5Hz
需累積最近 20 筆（約 6 秒）計算滑動窗口標準差
計分公式：score = (標準差 - 0.8) × 15 + 25，上限 49

標準差 0.81A → 25分 → WARNING 下緣　標準差 1.5A → 36分 → WARNING
標準差 2.5A+ → 上限 49分 → WARNING 上緣

為什麼這樣設計：軸承磨損是慢性劣化，不是緊急事故，設計上限 49 分讓它永遠在 WARNING，是預防性維護的訊號。頻率必須穩定才觸發，避免頻率變化本身造成的電流波動被誤判。窗口 20 筆讓標準差更穩定，寧可晚 6 秒發現也不要誤報。
啟動遮蔽

馬達開機時電流是額定的 1.5~2 倍，這是正常現象不是異常。main.py 在開機後 5 秒內不呼叫 Rules，避免把正常的高啟動電流誤判為緊急告警。

07 ML 判斷 — 隨機森林機器學習
什麼是監督式學習

機器學習有很多種，我們用的是「監督式學習」。核心概念：給機器看大量「有正確答案的例題」，讓它自己找出規律，之後遇到新題目就能判斷。

白話比喻
就像教小孩認動物：給他看 1000 張貓的照片說「這是貓」，再給他看 1000 張狗的照片說「這是狗」。學習完之後，看到新照片就能自己判斷。我們的 ML 也是這樣，只不過學的是「這是 STALL」「這是 BEARING_WEAR」。
訓練資料怎麼來

我們沒有真實的歷史異常資料，所以用 VFD_simulator 批次產生訓練資料。每種工況產生 1000 筆，五種工況共 5000 筆，每筆都附有正確答案（label）。

NORMAL × 1000 筆 → 每 20 筆重置頻率，確保覆蓋 30~60Hz 全範圍
OVERLOAD × 1000 筆 → 任何頻率下高電流的特徵
STALL × 1000 筆 → 低頻 + 極高電流的特徵
LOAD_LOSS × 1000 筆 → 任何頻率下極低電流的特徵
BEARING_WEAR × 1000 筆 → 先預熱 20 筆填滿窗口，確保 current_std 有意義
──────────────────────────────
總計 5000 筆，打亂後分割：
訓練集 4000 筆（80%）← 學習用
測試集 1000 筆（20%）← 考試用，模型沒看過
訓練集 vs 測試集：為什麼要分開

如果用同一批資料訓練又評估，就像用考古題練習然後用同一套考古題考試，100 分沒有意義。測試集是模型沒看過的資料，用它評估才知道模型真正學會了多少。

測試集準確率：100%，代表模型在模擬資料上能正確區分五種工況。

但要誠實說：訓練資料和測試資料都來自同一個 simulator，統計分布一致，這是 Simulation-to-Real 的限制，換真實硬體後準確率才是真正的考驗。

決策樹怎麼判斷

決策樹就像一連串的問題，每個問題把資料分成兩群，一層一層往下，最後走到葉子節點就是答案。

根節點：「電流超過 20A 嗎？」
├── 是 → 「頻率低於 35Hz 嗎？」
│　　　　├── 是 → STALL（低頻 + 高電流）
│　　　　└── 否 → OVERLOAD（正常頻率 + 高電流）
└── 否 → 「電流低於 5A 嗎？」
　　　　　├── 是 → LOAD_LOSS（負載流失）
　　　　　└── 否 → 「current_std 超過 0.8A 嗎？」
　　　　　　　　　　├── 是 → BEARING_WEAR（電流不穩）
　　　　　　　　　　└── 否 → NORMAL（正常）

每個節點的問題和切割點，是演算法從 6 個特徵裡自動找出「最能把資料分乾淨的條件」，不是人工設定的。

為什麼要 100 棵樹

一棵樹容易「記死答案」，對訓練資料學得太好，換一筆稍微不同的資料就可能判錯（過擬合）。

100 棵樹的解法：每棵樹訓練時隨機使用不同的資料子集和特徵組合，各自走不同的路，最後投票。就算幾棵樹判錯了，大多數還是對的。

樹001 主要看 current_a + torque_nm → 擅長判斷 STALL / LOAD_LOSS
樹002 主要看 slip_ratio + frequency_hz → 擅長判斷 STALL vs OVERLOAD
樹003 主要看 current_std + current_ratio → 擅長判斷 BEARING_WEAR vs NORMAL
...（共 100 棵，各有側重）

投票：STALL 87票 / OVERLOAD 13票 → 最終 STALL，信心 87%
六個輸入特徵

模型不直接看原始感測值，而是使用 6 個有物理意義的特徵，每個特徵都有其針對的異常：

01
current_a
電流絕對值，最直接的負載指標。STALL 極高、LOAD_LOSS 極低、OVERLOAD 偏高
02
torque_nm
轉矩，從功率和轉速推算。STALL 時轉矩極高，LOAD_LOSS 時極低，區分度大
03
slip_ratio
轉差率，從電流負載比估算。STALL 時轉差率高，搭配頻率能區分 STALL vs OVERLOAD
04
current_ratio
電流 / 額定電流，正規化的相對負載比，讓不同規格馬達的特徵更具可比性
05
frequency_hz
頻率，STALL 時拉不上去，是區分 STALL 和 OVERLOAD 的關鍵背景資訊
06
current_std
最近 20 筆電流的標準差，捕捉電流振幅。BEARING_WEAR 的唯一關鍵特徵，NORMAL 的標準差約 0.2A，BEARING_WEAR 約 1.4A
為什麼選隨機森林
✅ 小資料集（5000筆）表現好，神經網路需要更多資料
✅ 不需要特徵正規化，Hz、A、N·m 單位不同也沒關係
✅ 可解釋性強，可以輸出特徵重要性，知道模型在看什麼
✅ 不容易過擬合，100棵樹投票比單棵樹穩定
✅ 訓練快，幾秒完成，適合 demo 快速迭代

⚠️ 不擅長捕捉時間序列的長期趨勢（如電流緩慢爬升）
⚠️ 我們用 current_std 滑動窗口部分補償這個限制
08 雙層互補的價值
Rules 的強項

物理條件明確、特徵極端的異常，Rules 快速給出結論，原因完全透明可解釋。

STALL：電流 27A → score=70 → DANGER
LOAD_LOSS：電流 2A → score=64 → DANGER
立即告警，不需要等待
ML 的強項

Rules 看不到的三個場景，ML 靠多特徵組合補位：

輕度 OVERLOAD → Rules NORMAL / ML WARNING
STALL 低電流 → Rules OVERLOAD / ML STALL
BEARING_WEAR 初期 → Rules 等窗口 / ML 先補位
三個 ML 補位的具體場景

場景一：OVERLOAD 輕度（電流 17~19A）

電流 18A：
OVERLOAD score = (18-16.5) × 9 = 13分 → NORMAL（未到 WARNING 門檻）
ML 看電流比例 + 轉矩特徵 → 判 OVERLOAD WARNING
→ final = WARNING（ML 補位）

場景二：STALL 低電流模糊地帶（電流 20~24A）

電流 22A：
STALL score = (22-20) × 10 = 20分
OVERLOAD score = (22-16.5) × 9 = 49分 → 取最高，判 OVERLOAD WARNING
ML 看頻率低 + 轉矩高 → 判 STALL DANGER
→ final = DANGER（ML 升等）

場景三：BEARING_WEAR 前 20 筆（窗口未填滿）

第 1~19 筆：
Rules 標準差窗口未滿 → score=0 → NORMAL
ML 從電流振幅特徵 → 判 BEARING_WEAR WARNING
→ final = WARNING（ML 補位，提早約 6 秒發現）
綜合等級如何決定

取 Rules 等級和 ML 等級中較嚴重的那個，不會因為一層說正常就放過。

Rules = NORMAL / ML = WARNING（BEARING_WEAR 初期）
→ final = WARNING（ML 補位）

Rules = WARNING（OVERLOAD）/ ML = DANGER（STALL）
→ final = DANGER（ML 升等）

Rules = CRITICAL / ML = CRITICAL（STALL 電流 28A+）
→ final = CRITICAL（兩層確認，高度可信）
09 風險等級
正常
Score 0 ~ 24
設備運作正常，持續監控，無需採取行動
警告
Score 25 ~ 49
出現異常趨勢，安排檢查，持續觀察是否惡化
危險
Score 50 ~ 74
立即檢查設備，考慮降載或計劃停機檢修
緊急
Score 75 ~ 100
立即停機，繼續運轉可能造成設備損壞或人員安全事故
各工況對應的設計等級
STALL → 漸進式 DANGER / CRITICAL（Score 最高 100）← 立即停機
LOAD_LOSS → 危險（Score 50~74）← 盡快處理
OVERLOAD → 警告（Score 最高 49）← 安排檢查
BEARING_WEAR → 警告（Score 最高 49）← 預防性維護

各工況的分數上限刻意對應等級上緣，確保不同嚴重程度的異常永遠落在正確的風險區間。

10 系統誠實限制與未來展望
現有的限制（誠實說明）
⚠️
訓練資料來自模擬器（最重要的限制）
訓練和測試資料都由同一個 simulator 產生，測試集準確率 100% 是在模擬資料分布上的結果。換成真實硬體資料後，準確率可能下降，需要用真實資料重新訓練或微調。這是 Simulation-to-Real 開發流程的標準限制。
⚠️
轉差率和轉矩是估算值
沒有轉速感測器，slip_ratio 是從電流負載比推算的近似值，torque_nm 是用同步轉速計算的理論值。加裝編碼器後才能得到真實的轉速和精確的轉矩。
⚠️
BEARING_WEAR 是簡化版
真實的軸承磨損偵測需要高頻取樣（1000Hz 以上）和 FFT 頻譜分析（ISO 10816）。我們用電流標準差做簡化版的特徵提取，適合 demo 展示，真實環境需要升級。
⚠️
STALL 的 Rules 判斷在低電流區段靠 ML 補位
電流 20~24A 時 OVERLOAD 分數高於 STALL，Rules 判 OVERLOAD，由 ML 從頻率和轉矩特徵升等為 STALL。這是設計上刻意的分工，不是錯誤。
未來接軌真實硬體

系統架構已經為接真實硬體做好準備：

換真實硬體 → 只改 comms.py（實作 Modbus RTU）
換馬達規格 → 只改 config.py（銘牌規格和閾值）
重新訓練 ML → 用真實資料替換模擬訓練資料
升級 BEARING_WEAR → 加裝振動感測器 + FFT 分析

核心的雙層判斷架構、物理計算層、資料流設計都不需要改動。
